Bioinformatics Learning

Why have you learned the programming languages R and Python and use Linux OS (Ubuntu) distributions?

Working as an experimental wetlab and field biological technician, I came to the realization around 2012 that we were accumulating a lot of data! Gene expression data for just one experiment can occupy tens to hundreds of thousands of rows of genes/isoforms by hundreds of columns of samples. Too much data to work in Excel and perform calculations by hand.

Since then, I’ve devoted substantial time and effort towards learning Python and R and being comfortable working on the commandline in a Linux operating system or Mac to efficiently work with data sets of this size. I’ve become quite passionate about training others to do the same so that discoveries can be proportional to the data collected, since substantial investmetns are made in primary data generation. The software carpentry community is a great way to get started! In 2017, I became certified as an instructor.

I often hear this question. While there is benefit in learning how to code for the sake of building programming skills, we’re all busy. For students wanting to generally expand knowledge, Python and R both seem to be useful. For researchers who are pressed for time, language-learning should be project-driven. In my daily work, I use various software packages databases. Software packages serve diverse purposes, including sequence alignments, short read quantification, statistical analyses, bioinformatics database searching. Running these tools does not require knowledge of a programming language. I use the Python programming language to stitch together workflows, or “pipelines” as we call them so that the same pipeline can be run on multiple samples the same way. This ensures that samples are all treated equal and output is comparable. If more samples are added or a new software tool becomes available, programmable pipelines (not looping over all files in a directory) ensures efficient reproducibility of the work. If you’re working with large sets of samples or multiple species, programming pipelines has been a valuable time saver for me after running pipelines several times.

There tends to be a divide between software developers and the users. Principle Investigators with too much on their plate already can be easily persuaded to jump on a new software package bandwagon with the promise of saving time and efficiency. Being able to evaluate the output of running a software package is more valuable than spending time on programming. Is the output expected? Was there an error? What is really going on? Software developers are sometimes not as familiar with specific research applications than the users, who have generated the data and have a better first-hand knowledge of how the data were acquired. Communication between software developers and users is useful, and often necessary! Github is wonderful. If there are errors, it took me a long time to realize it is not the fault of the user. If the softare is open source and being developed on github, filing an issue is often very helpful for all parties to document a problem or question.

In the long-term, with Moore’s Law applied to gene sequencing (see below), the amount of accumulated data is going to soon (if not already) exceed many biologists’ current capabilities to store and process data. Many articles and books have been written encouraging biologists to learn computing skills for bioinformatics. Why not spend time and effort now on acquiring better computer skills to gain better control over the rapidly-growing amount of information later?
